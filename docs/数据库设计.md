# 数据库设计说明

本文档对应项目当前实现版本（`Spring Boot + MySQL`），用于论文“数据库设计”章节直接引用与二次整理。

## 1. 数据库与字符集

1. 数据库名建议：`traffic_warning`
2. 字符集建议：`utf8mb4`
3. 时区：`Asia/Shanghai`

## 2. 核心表清单

1. `sys_user`：系统用户（管理员/交通管理人员）
2. `accident_record`：交通事故记录
3. `warning_event`：预警事件主表
4. `warning_disposal_record`：预警处置记录
5. `warning_rule_config`：规则参数配置
6. `sys_operation_log`：系统操作审计日志
7. `ai_call_log`：大模型调用日志
8. `rule_change_log`：规则修改历史
9. `dict_item`：统一字典项（事故类型/道路类型/预警等级）

## 3. 关键表结构说明

## 3.1 `sys_user`

1. `id` BIGINT PK
2. `username` 登录名（唯一）
3. `password` BCrypt 密文
4. `display_name` 展示名
5. `role_code` 角色编码（`ROLE_ADMIN` / `ROLE_TRAFFIC_OFFICER`）
6. `enabled` 是否启用
7. `created_at` / `updated_at`

## 3.2 `accident_record`

1. `occur_time` 事故发生时间
2. `road_name` 道路名称
3. `area_code` 区域编码
4. `longitude` / `latitude` 经纬度
5. `accident_type` 事故类型（由字典 `ACCIDENT_TYPE` 约束）
6. `vehicle_count` 涉及车辆数
7. `casualty_count` 伤亡人数
8. `description` 事故描述
9. `created_by` / `created_at` / `updated_at`
10. `deleted` 逻辑删除标记

索引：
1. `idx_accident_time(occur_time)`
2. `idx_accident_road(road_name)`

## 3.3 `warning_event`

1. `warning_code` 预警编号（唯一）
2. `risk_level` 风险等级（`LOW/MEDIUM/HIGH`）
3. `target_road` 目标路段
4. `target_time_window` 统计时间窗
5. `trigger_reason` 触发原因文本（可追溯）
6. `status` 生命周期状态：
   `DRAFT -> CONFIRMED -> PUBLISHED -> PROCESSING -> RESOLVED`
7. `published_at` / `resolved_at`
8. `created_by` / `created_at` / `updated_at`

索引：
1. `idx_warning_status(status)`
2. `idx_warning_road(target_road)`

## 3.4 `warning_disposal_record`

1. `warning_id` 关联预警
2. `action_type` 动作类型
3. `action_desc` 处置说明
4. `operator_id` 操作人
5. `operated_at` 操作时间

索引：
1. `idx_disposal_warning(warning_id)`

## 3.5 `warning_rule_config`

1. `rule_key` 规则键（唯一）
2. `rule_value` 规则值（字符串形式存储）
3. `description` 规则说明
4. `updated_by` / `updated_at`

默认规则（系统初始化）：
1. `W_FREQ_7D=0.45`
2. `W_FREQ_30D=0.25`
3. `W_NIGHT=0.15`
4. `W_SEVERE=0.15`
5. `THRESHOLD_MEDIUM=40`
6. `THRESHOLD_HIGH=70`

## 3.6 `dict_item`

统一字典建模，避免多表重复：

1. `dict_type` 字典类型（如 `ACCIDENT_TYPE`）
2. `dict_code` 字典编码（如 `REAR_END`）
3. `dict_name` 字典中文名（如 `追尾`）
4. `sort_no` 排序号
5. `enabled` 启用状态
6. `remark` 备注
7. `updated_by` / `updated_at`

约束与索引：
1. 唯一索引：`uk_dict_type_code(dict_type, dict_code)`
2. 普通索引：`idx_dict_type_enabled(dict_type, enabled)`

## 3.7 审计与AI日志表

1. `sys_operation_log`：记录关键业务行为（含登录成功/失败）
2. `ai_call_log`：记录模型接口、成功标记、输入输出长度、错误信息
3. `rule_change_log`：记录规则调参前后值与操作时间

## 4. 表关系（逻辑）

1. `sys_user(1) -> accident_record(N)` 通过 `created_by`
2. `warning_event(1) -> warning_disposal_record(N)` 通过 `warning_id`
3. `sys_user(1) -> sys_operation_log(N)` 通过 `operator_id`
4. `sys_user(1) -> ai_call_log(N)` 通过 `operator_id`

说明：当前实现以业务完整与演示可控为优先，未强制设置所有外键约束，避免初始化和演示环境兼容问题。

## 5. 初始化策略

1. 启动时自动执行 `server/src/main/resources/db/schema.sql`
2. 自动初始化管理员账号与默认规则、默认字典项
3. 可选执行 `sql/init.sql` 注入示例事故数据

## 6. 论文可直接引用要点

1. 采用“主业务表 + 配置表 + 审计表 + AI日志表”分层设计
2. 预警闭环与规则可解释性通过 `warning_event` + `warning_rule_config` + `rule_change_log` 实现
3. 数据字典统一建模，支持后台动态维护并驱动业务校验
