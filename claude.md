# CLAUDE.md

本文件用于约束“城市交通事故预警系统”的后续协作开发行为，确保代码实现与毕业论文目标一致。

## 1. 项目上下文

1. 项目名称：城市交通事故预警系统
2. 论文方向：基于 `Spring Boot + Vue3` 的规则驱动事故预警平台
3. 核心目标：实用性、可维护性、业务闭环、可演示答辩
4. 非目标：不引入复杂深度学习模型，不做分布式超大规模架构

## 2. 技术栈约束

1. 后端：`Java 17`、`Spring Boot 3.x`、`MySQL 8`、`MyBatis-Plus`、`Spring Security + JWT`
2. 前端：`Vue3 + Vite + TypeScript + Pinia + Element Plus + ECharts`
3. 接口：HTTP JSON API（仅 `GET/POST`），统一响应结构
4. 地图：优先高德 JS API（如受限可替换 Leaflet）

## 3. 业务红线（必须遵守）

1. 必须实现预警闭环：`生成 -> 确认 -> 发布 -> 处置 -> 解除`
2. 风险评估必须是“规则可解释”而非黑盒算法
3. 每个关键操作必须有审计日志
4. 角色最少包含 `管理员` 与 `交通管理人员`
5. 所有状态变更必须校验合法流转，禁止任意跳转

## 4. 数据与安全规范

1. 密码必须加密存储（`BCrypt`）
2. 禁止拼接 SQL，统一参数化查询
3. 所有写操作记录 `created_by/updated_by`
4. 时间统一使用 `Asia/Shanghai`
5. 经纬度字段需做范围校验

## 5. 编码规范

1. 命名语义化，避免拼音缩写
2. Controller 只做参数接收和返回，不写复杂业务
3. Service 按业务模块拆分，避免“大而全”类
4. 公共异常统一处理，返回规范错误码
5. 单个函数建议不超过 80 行，超过需重构

## 6. API 设计规范

1. 统一前缀：`/api`
2. 仅允许 `GET` 与 `POST`：查询使用 `GET`，新增/更新/删除/状态流转使用 `POST`
3. 更新与删除建议使用动作式路径：`/update`、`/delete`
4. 分页参数统一：`pageNum`、`pageSize`
5. 统一响应结构示例：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

## 7. 风险评估实现原则

1. 评分公式与阈值来自配置表，允许后台动态调整
2. 默认按路段 + 时间窗统计（7 天、30 天）
3. 触发预警时保存“触发依据”文本，便于答辩解释
4. 算法更新不能破坏历史数据可追溯性

## 8. 前端实现原则

1. 页面必须覆盖：事故管理、预警中心、地图分析、统计报表、规则管理
2. 图表与表格数据口径必须一致
3. 关键页面需要空状态、加载状态、错误状态
4. 所有危险操作（发布/解除）必须二次确认

## 9. 测试与验收

1. 每个核心模块至少 1 组接口测试
2. 预警状态流转必须有专门测试用例
3. 提交前至少执行：
4. 后端：`mvn test`
5. 前端：`npm run build`
6. 验收以“能演示完整闭环 + 论文可解释”为标准

## 10. Git 与提交规范

1. 分支建议：`feature/*`、`fix/*`、`docs/*`
2. 提交信息格式：`type(scope): summary`
3. 常用 `type`：`feat`、`fix`、`refactor`、`docs`、`test`
4. 单次提交聚焦单一主题，避免混合无关改动

## 11. 推荐里程碑

1. `M1`：完成登录鉴权与基础字典
2. `M2`：完成事故管理与查询
3. `M3`：完成风险评估与预警生成
4. `M4`：完成预警闭环与日志审计
5. `M5`：完成可视化报表与答辩优化

## 12. 文档要求

1. 每个模块完成后同步更新接口文档
2. 关键设计（规则、状态机、权限）要有说明文档
3. 保留演示脚本：账号、操作路径、示例数据

---

如果后续新增功能与本文件冲突，以“毕业论文目标优先、可演示优先、业务闭环优先”为决策原则。
