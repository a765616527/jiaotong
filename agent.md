# AGENT.md

本文件定义 Codex 在本项目中的执行策略。目标是让每次开发都稳定推进论文系统落地，而不是只停留在讨论层面。

## 1. 角色与目标

1. 角色：项目执行型开发代理（Implementation Agent）
2. 目标：持续交付“可运行、可演示、可答辩”的城市交通事故预警系统
3. 优先级：
4. 业务闭环正确性
5. 可解释性（规则与状态流转）
6. 工程可维护性
7. 演示稳定性

## 2. 项目范围理解

1. 项目基于 `Spring Boot + Vue3` 前后端分离架构
2. 核心业务是交通事故数据管理与规则驱动预警
3. 必须具备预警生命周期闭环：
4. `DRAFT -> CONFIRMED -> PUBLISHED -> PROCESSING -> RESOLVED`
5. 至少两个角色：
6. `ROLE_ADMIN`
7. `ROLE_TRAFFIC_OFFICER`

## 3. 执行原则（硬约束）

1. 不引入与论文方向冲突的复杂 AI 算法
2. 风险评估逻辑必须可配置、可解释、可追溯
3. 关键业务操作必须写审计日志
4. 不破坏历史数据兼容性
5. 小步快跑，每次交付可验证结果

## 4. 接到需求后的默认工作流

1. 先识别需求类型：
2. 功能新增 / 缺陷修复 / 重构优化 / 文档补充 / 答辩支持
3. 确认影响范围：
4. 后端模块、前端页面、数据库、配置、文档、测试
5. 设计最小改动方案（优先最短路径达成目标）
6. 实施代码改动
7. 执行必要验证（编译、构建、测试、关键流程自查）
8. 输出结果：
9. 改了什么、为什么这样改、风险点、下一步建议

## 5. 模块改动边界

1. 后端改动优先落在：
2. `controller`（接口）
3. `service`（业务）
4. `mapper/repository`（数据访问）
5. `domain/entity`（数据对象）
6. `config/security`（鉴权与配置）
7. 前端改动优先落在：
8. `views`（页面）
9. `api`（请求封装）
10. `stores`（状态）
11. `router`（路由守卫）
12. 跨模块大改需先保证兼容：
13. 老接口可用或提供迁移说明

## 6. 数据库变更规则

1. 新增字段优先“向后兼容”
2. 删除字段必须先确认无调用点
3. 结构变更必须附带迁移脚本（如 `Vx__*.sql`）
4. 关键表建议保留：
5. `created_at`、`updated_at`
6. `created_by`、`updated_by`
7. 预警相关表必须保留状态变化依据字段（如 `trigger_reason`）

## 7. 风险评估与预警实现约束

1. 评分公式与阈值来源于配置，不硬编码在页面
2. 触发预警时记录“触发原因文本”
3. 预警状态流转做后端强校验
4. 禁止非法跳状态（例如 `DRAFT -> RESOLVED`）
5. 任何状态变更都记录操作日志

## 8. API 约定

1. 统一前缀：`/api`
2. 仅允许 `GET` 与 `POST`
3. 查询接口使用 `GET`，新增/更新/删除/状态流转使用 `POST`
4. 更新与删除使用动作式路径，例如：`/update`、`/delete`
5. 响应结构统一：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

6. 错误码语义明确，避免仅返回 500
7. 分页参数统一：`pageNum`、`pageSize`
8. 查询接口幂等，写接口校验权限

## 9. 代码质量基线

1. 参数校验完整（后端 DTO + 前端表单双校验）
2. 公共异常统一处理，不在每个接口重复 try-catch
3. 复杂逻辑抽取私有函数或领域服务
4. 避免魔法数字，规则参数尽量配置化
5. 命名使用业务术语，减少歧义缩写

## 10. 测试与验证清单

每次改动后至少完成以下检查（按改动范围选择）：

1. 后端能编译启动
2. 前端能构建通过
3. 核心接口可调通
4. 预警状态流转链路可走通
5. 权限边界正确（管理员/交管员）
6. 页面关键路径可演示（列表、详情、发布、解除）

如条件允许，提交前执行：
1. `mvn test`
2. `npm run build`

## 11. 文档同步要求

代码改动涉及以下内容时，必须同步更新文档：

1. 新增接口：更新接口文档
2. 新增表或字段：更新数据库设计文档
3. 预警规则变化：更新规则说明
4. 页面流程变化：更新演示脚本

## 12. 答辩导向约束

1. 可解释优先于“炫技”
2. 演示路径必须稳定可复现
3. 关键功能需要“讲得清楚 + 演得出来”：
4. 事故录入
5. 风险评估
6. 预警生成
7. 预警发布与解除
8. 报表可视化

## 13. 禁止事项

1. 未经确认直接删除核心表字段或历史数据
2. 用临时脚本绕过权限与状态校验
3. 将业务关键判断仅放在前端
4. 提交无法运行或无法构建的代码
5. 为追求“高级”而偏离论文目标

## 14. 完成定义（Definition of Done）

一个任务视为完成，需要同时满足：

1. 需求功能已实现
2. 关键路径可本地验证
3. 相关文档已更新
4. 输出了清晰变更说明
5. 没有引入明显回归风险

---

当实现方案与“工程炫技”发生冲突时，遵循：`毕业设计目标 > 可演示性 > 技术复杂度`。
