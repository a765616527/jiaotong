# 城市交通事故预警系统开发指南

## 1. 项目目标与范围

### 1.1 项目名称
城市交通事故预警系统（Graduation Design Edition）

### 1.2 目标
基于你在开题报告中的研究内容，建设一个可运行、可演示、可答辩说明的 Web 系统，重点实现以下能力：

1. 交通事故信息规范化管理（录入、编辑、查询、统计）
2. 事故风险识别（基于规则与阈值，不依赖复杂 AI 算法）
3. 预警闭环管理（生成、确认、发布、处置、解除）
4. 事故空间可视化与统计图表展示（地图 + ECharts）
5. 多角色权限控制（管理员、交通管理人员）

### 1.3 非目标（建议答辩时主动说明）
1. 不做实时流式大数据平台（如 Flink/Kafka 实时计算）
2. 不做深度学习事故预测模型
3. 不做城市级超大规模高并发生产部署

## 3. 总体架构设计

## 3.1 架构风格
采用前后端分离 + 分层单体架构（Monolith），保证毕业设计可控性与可交付性。

## 3.2 逻辑分层
1. `Controller`：处理 HTTP 请求、参数校验、统一响应
2. `Service`：核心业务逻辑（风险计算、预警流程）
3. `Mapper/Repository`：数据库访问
4. `Domain/Entity`：实体对象
5. `Infra`：日志、缓存、安全、任务调度

## 3.3 核心流程
1. 事故录入后进入事故库
2. 风险评估任务（手动触发或定时）按规则扫描
3. 触发条件满足则生成预警记录
4. 管理人员确认后发布预警
5. 处置完成后解除预警并记录全过程

## 4. 用户角色与权限

## 4.1 角色定义
1. `ROLE_ADMIN`：系统管理员
2. `ROLE_TRAFFIC_OFFICER`：交通管理人员

## 4.2 权限矩阵（最小可用版）

## 6. 业务流程设计

## 6.1 事故录入流程
1. 用户提交事故信息（时间、地点、类型、车辆数等）
2. 系统参数校验（必填、格式、经纬度范围）
3. 入库成功并写操作日志
4. 触发风险评估（同步或异步）

## 6.2 风险评估流程（规则驱动）
1. 按“路段 + 时间窗（近 7 天/30 天）”聚合事故数量
2. 计算风险分值
3. 对照阈值映射风险等级
4. 触发并生成预警

## 6.3 预警处置闭环
1. `DRAFT`（系统生成待确认）
2. `CONFIRMED`（人工确认）
3. `PUBLISHED`（已发布）
4. `PROCESSING`（处置中）
5. `RESOLVED`（已解除）

## 8. 风险评估规则设计（可解释、易答辩）

## 8.1 基础规则
使用加权评分法：

`risk_score = A * freq_7d + B * freq_30d + C * night_ratio + D * severe_ratio`

建议默认参数：
1. `A=0.45`
2. `B=0.25`
3. `C=0.15`
4. `D=0.15`

### 8.2 风险等级阈值示例
1. `score < 40`：低风险
2. `40 <= score < 70`：中风险
3. `score >= 70`：高风险（触发预警）

### 8.3 特殊规则（可选加分项）
1. 同一路段 24 小时内连续多起事故，直接提升一级风险
2. 极端天气期间阈值下调（如有天气数据接入）

## 20. 大模型 API 创新模块设计（本项目必做创新点）

### 20.1 设计原则
1. 大模型是“增强模块”，不替代核心规则引擎。
2. 预警触发判定仍由可解释规则完成，大模型负责“理解、表达、辅助决策”。
3. 所有大模型输出都应支持人工确认，不直接自动执行关键操作。

### 20.2 最推荐的 4 个落地点（按性价比排序）

1. 事故描述结构化抽取（强烈推荐）
2. 场景：事故录入时，管理人员输入自然语言描述（如执勤记录），大模型自动抽取结构化字段。
3. 目标字段：事故类型、可能责任因素、道路环境、是否夜间、是否恶劣天气、伤亡关键词等。
4. 价值：减少人工录入成本，提升数据标准化质量。

5. 预警原因自动解释（强烈推荐）
6. 场景：规则引擎触发预警后，大模型把“触发指标”翻译成人类易读说明。
7. 示例：把“近7天事故频次+夜间占比超阈值”解释为“该路段夜间事故集中，建议限速巡查”。
8. 价值：提升预警可理解性，便于管理决策与答辩展示。

9. 处置建议生成（推荐）
10. 场景：根据预警等级、事故类型、时段与历史处置记录，大模型生成“候选处置建议清单”。
11. 输出样式：`建议动作 + 适用条件 + 预期效果 + 风险提示`。
12. 价值：提高处置方案形成效率，但最终由人工确认。

13. 管理员智能问答（推荐）
14. 场景：在管理端增加“数据问答助手”，支持自然语言提问。
15. 示例问题：`本月事故最多的三个路段是什么？`、`夜间高风险路段有哪些共同特征？`
16. 价值：降低报表使用门槛，提升系统交互体验。

### 20.3 可选增强点（时间充足再做）
1. 周报/月报自动生成：基于统计结果自动产出“交通安全简报”草稿。
2. 多模态解析：对事故现场图片做辅助标签识别（若后续接入视觉模型）。

### 20.4 后端集成建议
1. 增加独立模块：`ai-service`（或 `LLMService`），统一封装模型调用。
2. 新增接口示例：
3. `POST /api/ai/extract-accident-fields`
4. `POST /api/ai/generate-warning-explanation`
5. `POST /api/ai/generate-disposal-suggestions`
6. `POST /api/ai/assistant-query`
7. 统一设置超时、重试、限流、熔断与降级（API 不可用时返回规则版兜底结果）。

### 20.5 Prompt 与输出约束
1. 采用模板化 Prompt，固定角色、任务、输出格式。
2. 强制 JSON 输出并做 Schema 校验，避免自由文本难以入库。
3. 对“建议类输出”增加免责声明：`仅供辅助，最终以人工研判为准`。

### 20.6 数据安全与合规要求
1. 发送给模型前进行脱敏（姓名、手机号、车牌号等个人敏感信息）。
2. 记录模型调用日志（时间、接口、token 消耗、响应时长、是否命中降级）。
3. 对外部模型 API 密钥使用环境变量管理，禁止硬编码到仓库。
